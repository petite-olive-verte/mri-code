# Architecture

How the module is built and deployed. If you just want to *use* `mri-code`, the
[skills reference](skills.md) is enough â€” this page is for understanding what the installer
does and why the target project ends up the way it does.

## Source-first

The repository is the **source**, not the thing you install:

- `payload/` â€” everything that gets deployed: `skills/`, `hooks/`, `mcp/`, `templates/`,
  `stacks/`, `constitution.md`, `models.md`, `settings.json`.
- The installer at the root copies a rendered subset of `payload/` into a target project.

Nothing is symlinked and nothing is generated by an LLM at install time â€” the target gets
plain files, and the same inputs always produce the same output.

## The installer

The installer is a single self-contained module, `mri_code_installer/main.py`:

- **No vendored engine, no external spec.** The deployment rules are inlined as plain
  Python constants and functions.
- **Deterministic copy.** Skills and hooks are copied as real files into `.claude/`, with a
  portable mirror under `.agents/skills/` for Codex.
- **Config substitution.** `--lang` / `--doc-lang` / `--user` are written to
  `.mri_code/config.json` and substituted into the content the installer renders. The
  agent reads that config at the start of every session.
- **Shared files stay yours.** `AGENTS.md`, `CLAUDE.md` and `.claude/settings.json` are written
  only if absent (an existing file is never touched), and `.mcp.json` is deep-merged (only the
  installer's own servers are added, never overwriting one already there). The manifest records
  what was created so `uninstall` reverses exactly that. See
  [installation.md â†’ Shared files](installation.md#shared-files-handled-non-destructively).

The `update` and `uninstall` subcommands reuse the same rules: `update` re-deploys and
prunes files dropped between versions; `uninstall` removes what was deposited. Neither
touches `.mri_code/docs/`.

## Runtime shape

Once installed, the module runs entirely inside the target project:

- **State on disk.** `.mri_code/docs/<project>/progress.md` (phases) and `plan.md`
  (fine-grained checkboxes) let a session stop and resume via `/mri-code-resume` â€” the
  flow does not rely on `/compact`.
- **Constitution.** `.mri_code/constitution.md` holds the quality bar, testing and architecture
  conventions the agent must respect. It ships **stack-agnostic**: the *Stack* section is an empty
  placeholder between two markers, and the scaffold skill seals the chosen stack into it from
  `.mri_code/stacks/<stack>.md`. The stack is a conclusion of `/mri-code-devplan`, not an install-time
  choice â€” brainstorm, spec and plan are written without one. The constitution is seed-once (yours to
  edit, never overwritten); the fragments under `stacks/` refresh on every update.
- **Visual feedback.** For web UIs the module uses MCP servers declared in `.mcp.json`:
  Playwright (drive/test) and Chrome DevTools (console/network/debug), in a
  write â†’ run â†’ observe â†’ fix loop.
- **Model routing.** Each step suggests a model per `.mri_code/models.md` (architecture and
  brainstorming â†’ Opus; code â†’ Sonnet; and so on).
